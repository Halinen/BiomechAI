# RAG 评估调查复盘  2026-05-06

> 第一次给项目搭 eval 流水线的完整记录：从「输出不理想」这个模糊抱怨，到定位根因、验证假设、撞到限流的全过程。

---

## 1. 起点：用户抱怨「输出不理想」

用户感受到 RAG 系统回答质量差，但说不清问题在哪。可能是检索没召回、可能是路由错层、可能是 LLM 幻觉、可能是 embedding 不适合中文——**完全没有 eval，无法定位根因**。

每次改动都靠主观感受判断好坏，优化方向盲目。

---

## 2. 假设：最可能拖累质量的环节

基于代码现状梳理出几个嫌疑：

| 嫌疑 | 证据 |
|---|---|
| **embedding 不匹配中文** | `EMBEDDING_MODEL = "all-MiniLM-L6-v2"` 是英文轻量模型，但用户输入是中文医学术语 |
| **相似度阈值边界过宽** | `SIMILARITY_ACCEPT=0.35 / SIMILARITY_REJECT=0.60`，按英文 cosine 调的；中文 query 距离落点未知 |
| **classify() prompt 过简** | 4 层定义靠自由文本解析，无 few-shot 示例 |
| **缺乏 eval 基础设施** | 上述三条都是猜测，没数据验证 |

**优先建 eval**——先用数据说话再决定改什么。

---

## 3. eval 设计

新增 `eval/` 模块，三个文件：
- [eval/testset.yaml](testset.yaml) — 15 条 mock 用例覆盖 7 个 collection
- [eval/eval.py](eval.py) — 主脚本
- [eval/eval_results.md](eval_results.md) — 自动产出的报告

**评估三段独立**：
- 路由：`classify()` 命中 `gold_layers`（IoU + 严格相等，程序计算）
- 检索：top-k chunk 是否覆盖 `gold_concepts`（LLM-as-judge）
- 生成：回答覆盖、幻觉、红旗处置（LLM-as-judge）

**用 Llama-3.3-70b 当 judge**：和生成同源，对幻觉检测会偏低估，但成本零、能跑就行。

---

## 4. baseline 跑完：暴露第一个根因

第一次完整跑 15 条用例（无翻译层）：

| 指标 | baseline |
|---|---|
| 路由严格命中 | 6.7% |
| 路由 Jaccard | 0.44 |
| **检索 concept recall@5** | **0.00** ← 关键 |
| 生成 concept 覆盖 | 0.42 |
| 幻觉发生率 | 66.7% |
| 生成总分 | 3.2/5 |

**所有 15 条用例的检索 recall 都是 0**——retriever 一个 chunk 都没召回。

去查 Chroma 原始距离分布：

```python
中文 query: "我做深蹲到底膝盖会内扣..."
distances: [0.872, 0.900, 0.911, 0.919, 0.927, ...]  # 全部 > 0.60 阈值

英文 query: "squat depth ankle dorsiflexion thoracic rotation"
distances: [0.424, 0.453, 0.462, 0.478, 0.485, ...]  # 正常区间
```

**根因实锤**：`all-MiniLM-L6-v2` 只懂英文，中文 query 的 embedding 跟英文 chunk 的 embedding 在向量空间里几乎不相关。当前 0.60 阈值对中文 query 等同于"全拒绝"。整个 RAG 退化为「无知识库直接 LLM 生成」——这就是用户感受到的「输出不理想」的根本原因。

---

## 5. 解决方案：query 翻译层（不是换 embedding）

### 用户的关键追问

> 「但是数据库都是英文的，不影响吗」

**这个问题问得对**——正因为知识库全是英文，翻译方案才合理：

```
当前：
  中文 query  ──[英文 embedding]──> 向量A（语义模糊）
  英文 chunk  ──[英文 embedding]──> 向量B（语义清晰）
  距离 = 0.87  ✗

翻译方案：
  中文 query → 翻译成英文 → [英文 embedding] → 向量A'（清晰）
  英文 chunk            → [英文 embedding] → 向量B（清晰）
  距离 ≈ 0.5  ✓
```

### 为什么不换中文/多语言 embedding

考虑过换 `bge-small-zh-v1.5` 或 `paraphrase-multilingual-MiniLM-L12-v2`：
- 优点：一劳永逸
- 缺点：要重建整个 Chroma 索引、重新校准阈值、引入新模型依赖

翻译方案的优势（基于「数据库是英文」这个前提）：
- 不重建索引（chunk 本来就是英文）
- 不换 embedding 模型
- 只在 [retriever.py](rag_system/retriever.py) 入口加一步「中→英翻译」，下游链路无感
- 医学/解剖术语翻译反而是 LLM 最擅长的（「膝盖内扣」→「knee valgus / knees caving in」）

### 实现

[rag_system/retriever.py](rag_system/retriever.py) 新增 `_translate_to_english()`：

```python
- 检测是否含中日韩字符（CJK），若不含直接返回原文
- 调 Llama 翻成英文医学/动作评估术语，max_tokens=120, temperature=0
- 用 dict 做 query 缓存（同一句中文每次只翻一次）
- 翻译失败兜底用原文
```

`query()` 入口加一行：
```python
search_text = self._translate_to_english(text)
query_vector = self._model.encode(search_text).tolist()
```

## 6. 验证：翻译方案有效

单条手动测试：

```
[Retriever] 翻译: '我做深蹲到底膝盖会内扣，脚后跟离地，胸椎也转不动...'
         → 'My knees internally rotate and my heels lift off the ground...'

distances: [0.495, 0.511, 0.525, 0.542, 0.542, ...]  # 全部进入 0.4-0.6 边界区
返回 5 个 chunk（之前是 0）
```

跑 eval（前 4 条 case 完整跑完后限流）：

| 指标 | baseline | 加翻译 |
|---|---|---|
| 检索 concept recall | **0.00** | **0.67** |
| case_01 路由错但检索召回 | recall 0.00 | recall 0.33 |
| case_02 (NASM 上交叉) | recall 0.00 | **recall 1.00** |
| case_03 (PRI 呼吸) | recall 0.00 | **recall 1.00** |
| case_04 (神经敏化) | recall 0.00 | recall 0.33 |
| 幻觉发生率 | 100%（4/4） | 50%（2/4） |

**翻译方案直接生效，从「全部 0」变成「平均 0.67」**。检索瓶颈实锤是 embedding 不匹配中文，被翻译层完美绕过。

## 7. 撞墙：Groq 免费档限流

跑完 case_04 之后：
```
RateLimitError: 429 - tokens per day (TPD): Limit 100000, Used 99989
Please try again in 4m2s
```

每日 100k token 配额用完。**为什么烧得这么快**：
- 翻译方案把所有 chunk 的距离从「全部 > 0.60 直接丢弃」推进到「全部在 0.35-0.60 边界区」
- 边界区每个 chunk 都触发一次 `_llm_judge` 调用（[retriever.py:84-105](rag_system/retriever.py)）
- 一次用户提问触发 30-45 次 judge 调用（query 多个 collection × 每个 collection 15 个候选）
- eval 还要在外层每个 chunk × 每个 concept 又调一次 judge——N×M 复杂度
- 估算单次 eval：~150k token，超出免费档

## 8. 暴露的衍生问题

eval 数据揭示了**翻译解决之后的新问题**：

### 问题 A：检索召回了知识，但生成 LLM 完全没用上

case_02：检索 recall=1.0（上交叉/胸大肌紧张/深层颈屈肌都检索到了），但生成 concept_coverage=0（回答里这些专业术语一个没提，全是「保持正确坐姿」「定期休息」这种通用话）。
→ 生成 prompt 没有强制 LLM 引用知识库内容，或 Llama-3.3-70b 在长 context 中倾向于忽略给定资料生成通用建议。

### 问题 B：路由严重过度触发

15 条 case 里 14 条 `classify()` 返回了「控制层、结构层、输出层」三层（甚至全四层）——LLM 倾向于「全选」而不是精准选层。
→ `classify()` prompt 过简，没有 few-shot 示例约束 LLM 选最相关的 1-2 层。

### 问题 C：边界区阈值不再适用

翻译后 chunk 距离从 0.87 变成 0.49——按当前阈值全部走 LLM judge 路径，慢且费 token。
→ 阈值需要按翻译后的实际分布重新校准（比如 ACCEPT 提到 0.50）。

## 9. 接下来的优先级

按性价比排序：

| 优先级 | 改动 | 预期效果 |
|---|---|---|
| **P0** | 收紧检索阈值（ACCEPT 0.35→0.50, REJECT 0.60→0.65） | 大部分 chunk 直接保留跳过 judge，每次 query 速度快 30 倍，token 消耗砍 80% |
| **P1** | 改进 `classify()` prompt，加 few-shot 例子 | 路由准确率从 6.7% 提升 |
| **P2** | 强化生成 prompt，强制引用 chunk 内容 | 解决「检索对了但回答没用上」 |
| **P3** | eval judge 批量化（N×M → 1 次每 case） | 跑 eval 更省 token，方便频繁迭代 |

## 11. 关键学习

1. **「输出不理想」是表层现象**，必须用 eval 把它分解为路由/检索/生成三段独立指标，才能定位根因。
2. **数据库语言决定 query 处理方式**——英文 KB + 中文 query，加翻译层比换 embedding 简单得多；如果是中文 KB，结论会反过来。
3. **解决一个瓶颈会暴露下一个**——翻译让检索 recall 从 0 变 0.67，但马上看到生成不引用 chunk、路由过度触发的新问题。eval 让这些问题可见。
4. **Llama-3.3-70b 当 judge 是权宜之计**，对幻觉检测系统性偏低估；关键决策前需 Claude 复核。
5. **免费 LLM 配额是真实约束**——eval 的 N×M 复杂度会 O(case×chunk×concept) 烧 token，要么收紧阈值、要么批量化、要么换付费档。
