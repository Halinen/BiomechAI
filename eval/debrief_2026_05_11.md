## 2. 路由优化：从 6.7% 命中升到至少局部 100%

### 改动前

```python
prompt = f"""你是动作评估专家。根据用户描述，判断问题主要涉及哪些层。
[四层定义...]
用户描述：{query}
只输出涉及的层名，用逗号分隔。"""
```

无 few-shot，无约束。结果：15/15 case LLM 返回 3-4 层全选。

### 改动后

```python
prompt = f"""你是动作评估专家。根据用户描述判断问题主要涉及哪 1-2 层。

【硬约束】
- 选最相关的 1-2 层。不要贪心全选。
- 描述很简单只涉及一个领域 → 只选 1 层。
- 仅当三层都有具体证据时才选 3 层（罕见）。

【示例】
描述："深蹲膝盖内扣，脚后跟离地"
→ {{"layers": ["结构层"], "reason": "都是关节活动度问题"}}

描述："长期伏案，头前伸，圆肩，做引体拉不起"
→ {{"layers": ["输出层"], "reason": "典型上交叉综合征代偿模式"}}

[共 7 个示例覆盖单层/双层/边界 case]

【现在判断】
描述："{query}"
只输出 JSON，不要解释："""
```

加 `response_format={"type": "json_object"}` 强制结构化输出。

### 测试结果（部分跑通的 case，trace 证据）

| Case | gold_layers | 新 classify 返回 | 评价 |
|---|---|---|---|
| case_01_fms_squat | `['结构层']` | `['结构层', '输出层']` | 比之前的「控制层+结构层+输出层」精准很多，多选了输出层（也合理） |
| case_02_nasm_uppercross | `['输出层']` | **`['输出层']`** | **完全命中**，对比 part1 的 0% → 100% |

LLM 还自己给出了 reason 字段（`"典型上交叉综合征代偿模式"`），证明 LLM 真的"想"了，不是模板化输出。

case_03 之后 token 耗尽走 fallback（`['结构层', '输出层']` 默认值），没有真实 classify 数据。

## 3. Observability：把 agent 决策路径变可观察

### 设计

[rag_system/trace.py](../rag_system/trace.py)：
- `start_trace(query, metadata)` 上下文管理器开启一次完整 query 的 trace
- `log_llm_call(component, prompt, response, model, extra)` 记录单次 LLM 调用
- 写 JSONL，按 trace_id 串联
- 通过环境变量 `RAG_TRACE_DIR` 启用，默认不写盘

5 个 LLM 调用点接入：
- `retriever.translate` (中→英翻译)
- `retriever.judge` (边界 chunk 相关性判断)
- `router.classify` (层分类)
- `agent.generate` (回答生成)
- `critic.review` (反思审查)

eval 自动包 trace：每条 case 一个 JSONL 文件。

### 实证产出

case_01 完整 trace（截取）：

```jsonl
{"type":"trace_start","trace_id":"4bd594b4","query":"我做深蹲到底膝盖会内扣..."}
{"step":1,"component":"router.classify","prompt_len":881,"response_len":79,
 "extra":{"detected_layers":["结构层","输出层"],"reason":"关节活动度受限 + 代偿模式"}}
{"step":2,"component":"retriever.translate","prompt_len":281,"response_len":116}
{"step":3,"component":"retriever.judge","extra":{"chunk_score":0.61,"kept":true}}
... (× 10 次 judge 调用)
{"step":14,"component":"agent.generate","prompt_len":1925,"response_len":599}
{"type":"trace_end","total_steps":14,"total_seconds":9.02}
```

### Trace 帮我们诊断了什么

按 component 统计 LLM 调用次数：

| case | router.classify | retriever.translate | retriever.judge | agent.generate | 总 |
|---|---|---|---|---|---|
| case_01 | 2 | 1 | **10** | 1 | 14 |
| case_02 | 2 | 1 | **12** | 1 | 16 |

发现 retriever.judge 占了 70%+ 的调用次数——这是 token 黑洞。后续改动 `ENABLE_LLM_JUDGE=False` 直接关掉，单 case 估算从 ~3.5k token 降到 ~1k。

**没有 trace 就看不到这个分布**——以前只能盯 stdout 计数，现在结构化的数据可以直接 grep / 聚合。

## 4. Token 经济学：免费档真实成本

### 量化今天烧光配额的过程

100k tokens/day 上限，今天的几次 eval 尝试：

| 时间 | 行为 | 烧 token |
|---|---|---|
| 09:00 | 第一次跑 --no-critic（旧 N×M judge）| ~30k |
| 12:00 | 改批量 judge 后单条 smoke | ~5k |
| 14:00 | sample 3 跑通 | ~10k |
| 16:00 | 尝试完整 15 条，挂在 case_03 | ~25k |
| 19:00 | 阈值收紧后再试，case_01-02 跑通挂 case_03 | ~25k |
| 20:00 | 关 judge 后再试，case_01 还是 429 | ~5k |
| **累计** | | **~100k** |

### 关键发现

1. **Groq 显示的 token 远多于我们用 char/3 估算的**——中文每 char 大概 1-2 token，加上 prompt structured 重复 character 的开销
2. **TPD 不是 24 小时一次性重置**——按分钟连续回流，但慢，今天一旦烧光基本就废了
3. **eval N×M judge 是隐藏成本**：每条 case 单纯 generate 只用 500-1000 token，但加上 judge 跳到 3.5k

### 各组件 token 消耗排序（基于 trace 数据）

1. `retriever.judge` × 10-12次/case：**最大头**，已禁用
2. `agent.generate`：1500-2500 token/case，固定开销
3. `router.classify` × 2次/case：~600 token，加 few-shot 后涨到 ~900
4. `retriever.translate` × 1次：~400 token
5. `critic.review` × 1-2次：~1500-3000 token（开 critic 时）

## 5. 真实学到的事

### 关于评估系统

- **eval 不只是评估工具，是诊断工具**：今天「拿不到对比数据」本身就是个发现——揭示了 token 经济学的真相，否则我们会一直以为「再调一调 prompt 就好」
- **Trace 比表格更有信息量**：报告里的「幻觉率 50%」是结果，trace 文件里的「为什么这条 case 触发了 10 次 judge」是过程，过程才是改进抓手

### 关于 agentic 设计

- **agent 越聪明，单 query 调用越多**：critic 反思一次 = 1 critic + 1 generate + 1 critic = 3 次大调用。免费档撑不住
- **Observability 是 agentic 系统的必需品**：单轮 RAG 不需要 trace（流程固定），agent loop 必须 trace（每条 query 路径都可能不一样）

### 关于工程权衡

- **`ENABLE_LLM_JUDGE=False` 是务实退让**：放弃边界 chunk 的精细判断，换 token 配额。这个 trade-off 在「向量距离已经够准」的前提下合理（翻译后大部分 chunk < 0.65 阈值直接保留）
- **简历可以讲「资源约束下的工程权衡」**：免费 LLM 配额是真实约束，怎么设计才能在配额内跑出有意义的 eval，比"我用最强模型"更显工程能力

## 8. 关键学习（继 Part 2）

6. **trace 是 agentic 系统的"X光"**：不可观察的 LLM 调用就是黑盒，trace 化之后 token 黑洞、决策错误都暴露出来
7. **Few-shot > 长定义**：classify prompt 长定义改了三次效果有限，加 7 个示例后立刻见效——LLM 学示例比学规则快
8. **`response_format={"type":"json_object"}` 是宝藏**：强制 JSON 输出后，再也没出现过解析失败
9. **资源约束是设计的一部分**：100k token/day 的硬约束逼着做 trade-off（关 judge、收紧阈值、batch judge），这些 trade-off 本身就是简历素材
10. **「拿不到数据」也是数据**：今天 5 次跑 eval 全部撞 429，这个事实本身让我们重新审视 architecture——发现 retriever.judge 是真正的 token 黑洞，引导出关键改动
